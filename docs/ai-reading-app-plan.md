# AI 读书应用方案草案

## 目标

做一个 Web 端的家庭读书应用，支持多人登录、统一公共书库、阅读进度独立保存，并能把上传的电子书自动切分、解析、朗读。

核心目标不是“能播音”，而是：

1. 断句正确
2. 朗读自然，偏故事讲述风格
3. 进度同步稳定
4. 全部尽量本地部署，不花钱

## 现阶段需求确认

- 仅 Web 端
- 普通家庭用户使用
- 公共共享书库
- 每个账户保留独立阅读进度
- 支持 `PDF / EPUB / TXT`
- 上传后自动解析、自动切分
- 朗读先不强调高级可控参数，后续再加音色、语速、情绪
- 允许在自己的电脑上部署
- 接受“时间换效果”，优先免费

## 推荐产品形态

这是一个“书库 + 阅读器 + 朗读引擎”三层结构：

1. 书库层：上传、管理、检索、标签、元数据
2. 阅读层：章节目录、正文展示、阅读进度、书签
3. 朗读层：分句、排版修复、TTS 播放、暂停续播

## 推荐技术路线

### 前端

- Web 应用
- 书籍详情页、阅读页、播放控制条、进度条、章节目录侧栏
- 朗读时正文高亮当前句

### 后端

- 用户认证
- 书籍上传与解析
- 文本切分与清洗
- 阅读进度保存
- TTS 请求调度
- 音频缓存

### 存储

- 数据库：保存用户、书籍、章节、句子、进度、书签
- 文件存储：保存原始电子书、解析后的文本、封面、生成音频

## 书籍解析建议

### EPUB

优先级最高。EPUB 结构天然适合做章节和段落解析，体验最好。

### TXT

最容易处理，但排版信息弱。适合直接做分段、分句和基础目录推断。

### PDF

最难处理。问题是：

- 版面复杂
- 换行不稳定
- 页眉页脚干扰
- 扫描版可能需要 OCR

建议策略：

1. 先支持文本型 PDF
2. 自动提取正文
3. 清理页眉页脚、页码、断行
4. 如果是扫描版，再走 OCR，作为后续增强

## 朗读质量建议

你最在意的是“像人”，所以不要把文本直接整段丢给 TTS。推荐流程：

1. 文本提取
2. 清洗与归一化
3. 章节拆分
4. 段落拆分
5. 句子级断句
6. 标点修复
7. 按句送入 TTS
8. 拼接播放

这样做的好处是：

- 停顿更自然
- 句读更准
- 高亮和播放同步更容易
- 后续可以插入情绪标签或角色标签

## 免费 / 本地部署方案建议

### 优先思路

优先选择可本地部署的开源 TTS，而不是依赖不稳定的免费云 API。

原因：

- 免费云接口通常有额度限制
- 音色和自然度一般
- 并发和稳定性不理想

### 本地 GPU 建议

如果你电脑上有 NVIDIA GPU，会更适合跑本地 TTS。

粗略判断：

- 8GB 显存：能跑一些轻量模型
- 12GB 显存：可选范围明显更大
- 16GB 及以上：体验更稳，速度更好

如果没有 GPU，也能做，但实时朗读体验会差一些，适合先离线生成音频，再播放缓存。

### 更现实的落地策略

先做“文本预处理 + 本地 TTS + 音频缓存”。

不要一开始就追求：

- 多角色对话式朗读
- 超强情绪控制
- 实时长篇连续生成

这些会显著提高复杂度。

## 局域网与访问方式

你说“看看网络啥的怎么搞”，我的建议是：

### 先做本机部署

- 你的电脑上启动 Web 服务
- 家里其他设备通过局域网访问
- 不需要公网暴露

### 访问方式

- 电脑本机：`http://localhost:xxxx`
- 同一局域网手机/平板：`http://你的内网IP:xxxx`

### 网络注意点

- 给你的电脑固定一个局域网 IP，避免每次变动
- 路由器和防火墙只放行局域网访问
- 不建议一开始直接暴露到公网

### 后续扩展

如果以后你想在外网访问，再考虑：

- 内网穿透
- 反向代理
- HTTPS
- 登录保护强化

## 外出听书访问方案

你希望短期在家里连 WiFi 听，后续在小区遛弯时也能打开网站继续听。这个需求可以分两步做。

### 第一阶段：家里 WiFi 使用

这是最简单、最稳的方式。

部署方式：

- 你的电脑运行服务端
- 手机、平板、电脑连接同一个家里 WiFi
- 家人用浏览器打开内网地址

访问形式大概是：

```text
http://你的电脑内网IP:端口
```

优点：

- 不需要买服务器
- 不需要公网 IP
- 不需要复杂网络配置
- 开发和调试都简单

缺点：

- 离开家里 WiFi 后访问不了
- 家里电脑需要开机

### 第二阶段：外面打开网站使用

如果希望家人在小区、路上、外面直接打开网站听，推荐使用 Cloudflare Tunnel。

它的目标效果是：

```text
https://read.your-domain.com
```

家人的使用方式：

1. 打开网址
2. 登录账号
3. 继续听书

家人不需要：

- 安装 Tailscale
- 安装 ZeroTier
- 配置 VPN
- 理解网络设置

### Cloudflare Tunnel 是什么

可以把它理解成：

你的电脑主动连到 Cloudflare，Cloudflare 再给外面的人一个安全网址访问你的电脑服务。

好处是：

- 不需要家里宽带有公网 IP
- 不需要路由器端口转发
- 不需要把家里电脑直接暴露到公网
- 可以使用 HTTPS
- 外部访问体验就是普通网站

### Cloudflare Tunnel 要不要钱

家庭自用场景下，Cloudflare Tunnel 通常可以免费使用。

可能需要花钱的是域名。

如果你想让家人访问一个固定好记的网址，例如：

```text
https://read.xxx.com
```

通常需要买一个域名。便宜域名一般一年几十元左右。如果你已经有域名，就不用再买。

如果完全不想买域名，也可以先用临时地址测试，但临时地址不稳定，不适合长期给家人使用。

### 不推荐一开始用 Tailscale / ZeroTier 的原因

Tailscale 和 ZeroTier 技术上很好，也比较安全，但对家人不够简单。

它们通常要求：

- 手机安装客户端
- 登录账号
- 加入你的私有网络
- 后续还要保持客户端在线

如果只是你自己用，可以考虑。

如果目标是让家人“打开网站就能听”，Cloudflare Tunnel 更合适。

### 是否需要 App

短期不建议直接做 App。

原因是这个项目真正复杂的地方在：

- 书籍解析
- 分句
- TTS 朗读质量
- 音频缓存
- 阅读进度同步

先把 Web 版跑通更重要。

推荐路线：

1. 第一版：Web，家里 WiFi 使用
2. 第二版：Web + Cloudflare Tunnel，外面打开网站使用
3. 第三版：PWA，手机可以添加到主屏幕，体验接近 App
4. 第四版：如果确实需要离线缓存、锁屏控制、通知栏播放，再考虑 App

### 推荐网络路线

综合你的目标，推荐这样走：

1. 先做本地 Web 应用
2. 家里 WiFi 访问验证功能
3. 功能稳定后加登录保护
4. 买一个便宜域名，或使用已有域名
5. 配置 Cloudflare Tunnel
6. 家人外出时通过固定网址访问

这条路线对家人最简单，对你来说也比买云服务器和配置公网 IP 更轻。

## 数据设计建议

建议至少有这些核心实体：

- 用户
- 书籍
- 书籍文件
- 章节
- 段落 / 句子
- 阅读进度
- 书签
- 播放记录

### 进度保存

阅读进度不要只存“第几页”，因为 PDF、EPUB、TXT 的页面概念不统一。

建议存：

- 书籍 ID
- 用户 ID
- 当前章节
- 当前句子或段落锚点
- 播放位置
- 更新时间

这样不同格式更通用。

## MVP 范围建议

第一版只做这些：

1. 用户登录
2. 上传 EPUB / TXT / PDF
3. 自动解析书籍结构
4. 书籍详情页
5. 阅读页
6. 分句朗读
7. 阅读进度保存
8. 本地 TTS 播放

第一版先不要做：

- 社交功能
- 评论
- 多设备复杂同步
- 高级情绪调参面板
- 公网开放

## 风险点

### PDF 解析不稳定

这是最大风险。需要接受“不同 PDF 效果差别很大”。

### TTS 断句质量

如果文本清洗做得不好，朗读会很机械。

### 本地算力

如果机器性能不足，长篇实时朗读会卡。

### 音频缓存

不缓存就会重复计算，体验差且浪费算力。

## 建议的实施顺序

1. 先定技术栈
2. 先做用户体系和书库
3. 再做 EPUB/TXT 解析
4. 再补 PDF
5. 再接本地 TTS
6. 再做阅读高亮与进度同步

## 我对你的建议

如果你目标是“给家人长期用”，最稳的路线是：

- 本地部署
- 局域网访问
- EPUB 优先
- PDF 作为增强
- 朗读走句子级调度
- 音频做缓存

这样最接近“能长期稳定用”，也最符合“不花钱、时间换效果”的目标。
